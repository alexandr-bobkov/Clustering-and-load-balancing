## Как работает эта связка c keepalived:

1. Мониторинг: Keepalived на «Мастере» постоянно проверяет (актуально если 2 и более сервера с HAProxy): «Работает ли мой локальный HAProxy?».
2. Реакция: Если скрипт возвращает ошибку (HAProxy упал), Keepalived понимает, что этот сервер больше не годен для работы.
3. Переезд IP: Он передает  Virtual IP второму серверу (Backup) c HAProxy.
4. Результат: Пользователи продолжают заходить на тот же IP, но попадают уже на второй HAProxy, который «живой».
- **У HAProxy и Keepalived  разные специализации:**
- Keepalived — это «грубый» инструмент. Он умеет только переключать IP-адрес с одной машины на другую (Failover). Он не умеет балансировать трафик между 10 разными серверами одновременно.
- HAProxy — это «тонкий» инструмент. Он умеет анализировать трафик, разделять его по весам, следить за сессиями пользователей и балансировать нагрузку (Load Balancing).

frontend http_frontend

    bind *:8888

backend weighted_servers
    # Режим должен совпадать с frontend

## ОТВЕТ:

- Скрипт необходимо установить на каждом сервере (файл кидаю в /etc/keepalived/update_priority.sh) и добавить на выполнение через crontab:

```bash
#!/bin/bash

# --- НАСТРОЙКИ ПУТЕЙ И ПЕРЕМЕННЫХ ---

# Указываем путь к итоговому файлу, который будет читать Keepalived
TRACK_FILE="/etc/keepalived/vrrp_priority"

# Указываем путь к временному файлу (нужен для безопасной "атомарной" записи)
TEMP_FILE="/etc/keepalived/vrrp_priority.tmp"

# Устанавливаем базовое число приоритета для абсолютно свободного сервера
# VRRP позволяет значения от 1 до 254. 250 — отличный запас.
BASE_PRIO=250

# --- ПОЛУЧЕНИЕ ДАННЫХ О НАГРУЗКЕ ---

# Команда uptime выдает строку с Load Average. Мы вырезаем первое число (за 1 минуту).
# awk разделяет строку по метке 'load average:', берет правую часть, а cut забирает число до первой запятой.
LOAD=$(uptime | awk -F'load average:' '{ print $2 }' | cut -d, -f1 | xargs)

# --- ПРЕВРАЩЕНИЕ ТЕКСТА В ЧИСЛО ---

# Bash не умеет работать с дробями (0.35), поэтому мы убираем точку: "0.35" -> "035"
# Команда tr -d '.' просто стирает символ точки.
LOAD_WITHOUT_DOT=$(echo "$LOAD" | tr -d '.')

# Если в начале числа остались нули (например, "035"), Bash может запутаться.
# Команда sed 's/^0*//' удаляет все нули в самом начале строки: "035" -> "35"
LOAD_NUM=$(echo "$LOAD_WITHOUT_DOT" | sed 's/^0*//')

# Если нагрузка была 0.00, то после удаления точек и нулей переменная станет пустой.
# Эта проверка [ -z ... ] ставит 0, если в переменной пусто, чтобы расчет не сломался.
[ -z "$LOAD_NUM" ] && LOAD_NUM=0

# --- РАСЧЕТ ИТОГОВОГО ПРИОРИТЕТА ---

# Вычитаем очищенное число нагрузки из базового приоритета.
# Пример: 250 (база) - 35 (нагрузка 0.35) = 215 (новый приоритет)
NEW_PRIO=$((BASE_PRIO - LOAD_NUM))

# --- ПРОВЕРКА ГРАНИЦ VRRP ---

# По стандарту VRRP приоритет не может быть меньше 1.
if [ "$NEW_PRIO" -lt 1 ]; then
    NEW_PRIO=1
fi

# По стандарту VRRP приоритет не может быть больше 254.
if [ "$NEW_PRIO" -gt 254 ]; then
    NEW_PRIO=254
fi

# --- БЕЗОПАСНАЯ (АТОМАРНАЯ) ЗАПИСЬ ---

# Сначала записываем итоговое число во временный файл.
echo "$NEW_PRIO" > "$TEMP_FILE"

# Мгновенно заменяем рабочий файл временным. 
# Это гарантирует, что Keepalived не прочитает файл в момент его перезаписи.
mv "$TEMP_FILE" "$TRACK_FILE"
```

- Настройка crontab на каждом сервере:
```bash

```

<summary>Шпаргалка по логам:</summary>

- Анализ логов Keepalived при работе с vrrp_track_file

- При использовании динамического изменения приоритета через файл, в системном логе (journalctl -u keepalived) фиксируются следующие критически важные этапы работы кластера:

### 1. Инициализация и чтение файла
При запуске службы Keepalived считывает начальное значение из указанного в конфиге файла.
*   **Лог:** `(VI_1) Changing effective priority from 100 to 254`
*   **Значение:** Это подтверждает, что механизм `track_file` активен. Keepalived игнорирует статичное значение `priority 100` из конфига и подставляет актуальное число `254` из  Bash-скрипта.

### 2. Динамическое изменение приоритета (Реакция на нагрузку)
Когда нагрузка на сервере растет, Bash-скрипт (по cron) перезаписывает файл меньшим числом.
*   **Лог:** `(VI_1) Changing effective priority from 254 to 150` (или до `101`, `50` и т.д.)
*   **Значение:** Параметр `effective priority` — это текущая "сила" сервера в сети. Как только это число падает, сервер становится кандидатом на переход в режим ожидания (BACKUP).

### 3. Сравнение приоритетов между узлами (Advert)
Серверы постоянно обмениваются VRRP-пакетами (Adverts). Если текущий MASTER видит соседа, который в данный момент "сильнее", происходит сравнение.
*   **Лог:** `Master received advert from 192.168.32.130 with higher priority 254, ours 150`
*   **Значение:** Ваш сервер (MASTER) признает, что у соседа (`.130`) приоритет выше. Это прямое условие для инициации переезда виртуального IP.

### 4. Переход в состояние BACKUP (Переезд IP)
После того как сервер признал превосходство соседа по приоритету, он добровольно отдает роль лидера.
*   **Лог:** `(VI_1) Entering BACKUP STATE`
*   **Значение:** В эту секунду виртуальный IP-адрес удаляется с интерфейса текущего сервера и перехватывается сервером с самым высоким приоритетом (тем самым "менее нагруженным").

### 5. Возврат в состояние MASTER (Восстановление)
Когда нагрузка на сервере падает, скрипт возвращает в файл число `254`. Если этот приоритет становится выше, чем у текущего временного лидера, сервер забирает IP обратно.
*   **Лог:** `(VI_1) Entering MASTER STATE`
*   **Лог:** `(VI_1) setting VIPs.`
*   **Значение:** Система вернулась в исходное состояние, так как данный узел снова стал самым свободным в кластере.

---
**Итог:** Данная последовательность логов показывает полную автоматизацию процесса: от мониторинга ресурсов скриптом до принятия решения протоколом VRRP о миграции IP-адреса.
